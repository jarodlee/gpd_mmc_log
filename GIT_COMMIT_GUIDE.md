一、为什么要规范提交？
良好的提交记录可以：
清晰记录每一次改动
方便问题回溯
提高团队协作效率
训练工程化思维
自动生成更新日志（Changelog）
提交历史 = 项目的“工程成长记录”。
二、提交格式规范
每次提交必须遵循以下结构：
<类型>: <一句话简要说明>

【变更说明】
- 做了什么
- 修改了哪些文件或模块

【变更原因】
- 为什么要修改
- 解决了什么问题

【测试情况】
- 是否测试
- 是否影响旧功能
三、提交类型说明
类型	说明	使用场景
feat	新功能	新增页面、新增脚本
fix	修复问题	修复 bug
docs	文档修改	修改 README
refactor	代码重构	优化结构但不改变功能
perf	性能优化	提高运行效率
style	格式调整	不影响逻辑的格式修改
test	测试相关	添加或修改测试
chore	杂项维护	更新依赖、配置调整
四、示例（学生课堂版）
feat: 增加日志网页展示功能

【变更说明】
- 新增 mmc.html 页面
- 新增 json 数据接口

【变更原因】
原始日志不方便阅读，需要可视化展示

【测试情况】
浏览器测试正常
五、示例（团队协作专业版）
refactor: 优化日志统计模块结构

【变更说明】
- 重构 mmc_logger.sh 逻辑
- 拆分统计函数
- 增强异常处理

【变更原因】
原逻辑耦合度过高，不利于扩展

【测试情况】
连续运行48小时无异常
六、提交原则
一次提交只做一件事（单一职责）
提交说明必须可读
不允许使用模糊描述：
update
修改一下
改改
test
每次提交都要写清楚“为什么”
七、课堂简化版本
适用于低年级或首次接触 Git 的学生：
<类型>: <一句话说明>

- 做了什么
- 为什么
八、启用提交模板
创建模板文件：
~/.gitmessage.txt
内容：
<类型>: <简要说明>

【变更说明】
- 

【变更原因】
- 

【测试情况】
- 
启用：
git config --global commit.template ~/.gitmessage.txt
九、进阶建议
长期团队协作可升级为：
使用 Conventional Commits 规范
自动生成 CHANGELOG
使用 Git Hooks 校验提交格式
使用 CI 自动检测规范
结合语义化版本（Semantic Versioning）

工程化思维
协作能力
